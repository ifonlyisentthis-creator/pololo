# Polarity V3 Implementation Plan

## Part 1: Bug Fixes (9 bugs)

All bugs should be fixed BEFORE starting the theme system, since some affect the same state management code that the theme system will extend.

### Bug 1: Stale isNearHighScore/isInRecordTerritory after revive

**Root cause**: In `revive()` (game_engine.dart line 231-234), `isNearHighScore` and `isInRecordTerritory` are reset to false. However, `highScore` was already updated in `_die()` (line 642-644) to match or exceed the current score. After revive, the player's score is unchanged, and the very first obstacle scored calls `_updateNearHighScore()` (line 1020), which immediately recalculates `isNearHighScore = score >= highScore - 5`. Since `score == highScore` after the first death updated it, the tension indicators fire instantly with no dramatic buildup.

**Fix**: In `revive()`, set a `_scoreAtRevive` field to the current score. In `_updateNearHighScore()`, only activate the near-high-score system when `score > _scoreAtRevive`. This means the player must actually surpass their pre-revive score before tension effects kick in.

```
// In revive():
_scoreAtRevive = score;

// In _updateNearHighScore():
if (highScore <= 10 || score <= _scoreAtRevive) {
  isNearHighScore = false;
  isInRecordTerritory = false;
  return;
}
```

### Bug 2: Cross-mode tier inflation

**Root cause**: In `_die()` (game_engine.dart lines 657-664), `GameConstants.getTier(highScore, easyMode: easyMode)` uses the CURRENT mode's thresholds, but `highScore` might have been set in a different mode. Easy mode thresholds are much higher (80/160/320/560/800 vs 50/100/200/350/500), so a score of 350 set in easy mode evaluated against hard thresholds would grant Diamond tier (350 >= 350) instead of Silver (350 >= 320 in easy).

**Fix**: Store the mode in which the high score was achieved. Two approaches:

Option A (Minimal, recommended): Add a `_highScoreMode` field to the engine and persist it in storage. In `_die()`, when setting a new high score, also record the mode. For tier calculation, always use the mode that corresponds to the high score:

```
// In _die(), when highScore is updated:
if (score > highScore) {
  highScore = score;
  _highScoreMode = easyMode;
}
// For tier:
final newTier = GameConstants.getTier(highScore, easyMode: _highScoreMode);
```

Option B (Complete, but bigger change): Store separate high scores per mode. This is cleaner but requires more storage and UI changes. Recommend Option A for V3.

### Bug 3: tierJustUnlocked fires twice on revive

**Root cause**: When the player dies the first time with a score that unlocks a new tier, `tierJustUnlocked = true` is set in `_die()` (line 662). The death screen shows the tier badge, and `_onDeath()` in game_screen.dart persists the tier and plays the tier_up sound (lines 157-162). Then `revive()` resets `tierJustUnlocked = false` (line 234). But when the player dies again, `_die()` runs again with `score == highScore` (line 657 condition is true), recalculates `newTier = getTier(highScore)`, and since `currentTier` was already updated in the first death, `newTier == currentTier`, so `tierJustUnlocked` should NOT fire again... UNLESS the bug is actually that `revive()` does NOT reset `previousTier` to match `currentTier`, so the death screen shows stale data.

Deeper analysis: The more likely path is that `_onDeath()` in game_screen.dart checks `_engine.tierJustUnlocked` (line 157) and persists it. Then revive sets it false. Then second death: `_die()` checks `score == highScore` which is true (score hasn't changed), recalculates tier, `newTier == currentTier`, condition `newTier > currentTier` is false, so `tierJustUnlocked` stays false. The second death screen would show `tierJustUnlocked = false`. This seems correct.

However, the edge case: If the player scores PAST another tier threshold after revive, then `newTier > currentTier` would be true, and `tierJustUnlocked` fires again legitimately. But `_onDeath()` also fires `tier_up` audio and persists again.

The actual bug is likely that `_onDeath()` is called, which persists `tierJustUnlocked` state, but the death screen is shown 800ms later asynchronously (line 178). If the player is fast enough to trigger revive before the death screen fully appears (edge case in rapid tap flow), `tierJustUnlocked` could be consumed inconsistently.

**Fix**: Add a definitive consumption pattern. In `_onDeath()`, immediately capture and clear `tierJustUnlocked`:

```
// In _onDeath():
final tierUp = _engine.tierJustUnlocked;
_engine.tierJustUnlocked = false; // consume immediately
if (tierUp) {
  storage.setMilestoneTier(_engine.currentTier);
  ...
}
```

Pass `tierUp` as a captured value to the death screen instead of reading it from the engine 800ms later. Also reset `tierJustUnlocked = false` in `revive()` (already done, but ensure the consumption pattern is consistent).

### Bug 4: isNewHighScore fires true after revive with no score improvement

**Root cause**: In `_die()` (line 642), when `score > highScore`, `_newHighScoreThisRun = true` is set and never cleared. In `revive()` (lines 219-242), `_newHighScoreThisRun` is NOT reset. So after revive, when the player dies again, even with the same score, `isNewHighScore` (line 1034: `bool get isNewHighScore => _newHighScoreThisRun`) returns true. This causes `_onDeath()` in game_screen.dart to re-persist the high score and the death screen to show "NEW BEST" again.

**Fix**: Either (a) reset `_newHighScoreThisRun = false` in `revive()`, or (b) use a more nuanced approach: track the high score BEFORE the run started and only show "NEW BEST" if the final score exceeds it. Approach (a) is simpler and correct:

```
// In revive():
_newHighScoreThisRun = false;
```

Then if the player scores past the high score AGAIN after revive, `_die()` would naturally re-set `_newHighScoreThisRun = true` because the condition `score > highScore` would be true (highScore was set to the previous death's score, so they need to exceed it).

### Bug 5: isTouching not reset in startGame()

**Root cause**: Actually, `isTouching = false` IS set in `startGame()` at line 164. But the real issue is that `isTouching` is driven by the `Listener` widget in game_screen.dart (lines 270-275): `onPointerDown` sets `isTouching = true`, `onPointerUp/Cancel` sets it false. If the player's finger is ON the screen when they tap "RESTART" on the death screen, the death screen pops, the Listener in the game screen receives the pointer-down event, and `isTouching` gets set back to true immediately after `startGame()` clears it.

Additionally, `isTouching` is NOT reset in `revive()` (lines 219-242). If the player was touching when they died, `isTouching` remains true during countdown, and on resume the ball immediately pulls right.

**Fix**: Reset `isTouching = false` in `revive()`. For the `startGame()` race condition with the Listener, add a brief input lockout: set a `_inputLockoutTimer` in `startGame()` that ignores touch input for the first ~100ms, or reset `isTouching = false` at the start of the first `_updatePlaying()` call after `startGame()`.

```
// In revive():
isTouching = false;

// In startGame():
isTouching = false; // already there
_inputLockoutFrames = 3; // ignore first 3 frames of input
```

### Bug 7: DST streak calculation edge case

**Root cause**: Looking at `updateStreak()` in storage_service.dart (lines 85-110), the code constructs `yesterdayDate = DateTime(today.year, today.month, today.day - 1)` which correctly handles calendar day arithmetic. However, there is a remaining edge case: `DateTime.now()` returns local time. During a DST "fall back" transition, the local clock repeats an hour. If a player plays at 1:30 AM the first time (before fall back), the date is recorded as today's date. Then DST kicks in, clock goes back to 1:00 AM. The player plays again. `DateTime.now()` returns the same calendar day, `todayStr == lastDate`, so streak is preserved correctly.

The actual remaining bug is likely: the `DateTime(today.year, today.month, today.day - 1)` constructor creates a midnight DateTime. If `today` is January 1 and the local timezone has a DST transition near midnight on December 31, the constructed DateTime might normalize to an unexpected date in rare timezone configurations. However, Dart's `DateTime` constructor handles `day: 0` by going to the last day of the previous month, so `DateTime(2024, 1, 0)` correctly becomes December 31, 2023.

A more practical edge case: If the device timezone changes (e.g., travel), the date string comparison could break. A player could play at 11 PM in timezone A (Jan 5), travel to timezone B where it's still Jan 4, play again. `todayStr` is "2024-01-04" but `lastPlayDate` is "2024-01-05". This is > yesterday, and != today, so streak resets to 1. This is arguably incorrect.

**Fix**: Use UTC for all date comparisons to eliminate timezone/DST ambiguity:

```
final today = DateTime.now().toUtc();
```

Or accept the current behavior and document it. Since this is a minor edge case, the simpler fix is to use `DateTime.now()` but compare using `difference.inDays` after normalizing both to noon:

```
final todayNoon = DateTime(today.year, today.month, today.day, 12);
final lastNoon = DateTime(lastYear, lastMonth, lastDay, 12);
final diff = todayNoon.difference(lastNoon).inDays;
```

### Bug 9: Player position not re-centered on revive

**Root cause**: `revive()` (lines 219-242) resets player alive state and velocity but does NOT reset `player.x`. The player stays at whatever x position they died at, which could be right against a wall. ```
// In revive():
player.x = screenWidth / 2;
player.y = screenHeight * playerYFraction;
player.velocityX = 0;

Also clear nearby obstacles that would immediately collide:

```
// Remove obstacles within a dangerous zone around the player
obstacles.removeWhere((obs) {
  final dist = (obs.worldY - player.y).abs();
  return dist < obstacleSpacing * 0.5;
});
```

  final List<Color>? invertedBallColors;

  final List<Color>? invertedObstacleColors;

  const VisualTheme({ ... });

  /// Auto-invert colors for Phase 5 white background
  VisualTheme withInversion() {
    if (invertedBallColors != null) return this; // already has overrides
    return VisualTheme(
      // ... copy all fields, but invert any too-bright colors
      ballColors: ballColors.map(_ensureContrastOnWhite).toList(),
      trailColors: trailColors.map(_ensureContrastOnWhite).toList(),
      // etc.
    );
  }

  static Color _ensureContrastOnWhite(Color c) {
    final luminance = c.computeLuminance();
    if (luminance > 0.7) {
      // Too bright for white background, darken
      return HSLColor.fromColor(c).withLightness(0.3).toColor();
    }
    return c;
  }

    [/* 10 tier 3 themes */],
    [/* 10 tier 4 themes */],
    [/* 10 tier 5 themes */],
  ];

  /// Select theme for a given score milestone.
  /// rotationIndex is persisted per-tier so each game shows a different variation.
  static VisualTheme? selectTheme(int score, Map<int, int> rotationIndices) {
    final tier = _scoreToTier(score);
    if (tier == 0) return null; // classic look

    final variations = themes[tier];
    final idx = (rotationIndices[tier] ?? 0) % variations.length;
    return variations[idx];
  }

  static int _scoreToTier(int score) {
    if (score >= 500) return 5;
    if (score >= 400) return 4;
    if (score >= 300) return 3;
    if (score >= 200) return 2;
    if (score >= 100) return 1;
    return 0;
  }

  /// Advance the rotation index for a tier (called when theme activates)
  static int advanceRotation(int tier, int currentIndex) {
    return (currentIndex + 1) % 10;
  }
}

## Part 3: Parameterized Rendering System

### 3A: ball_painters.dart -- Ball Shape Drawing Functions (~700-900 lines)

This file contains one static method per BallShape enum value. Each method takes `Canvas`, `double radius`, `List<Color> colors`, `double glowPhase`, `double intensity` and draws the shape at origin (caller handles canvas translate/save/restore).

Key implementation details for each shape:

- **circle**: Existing `_drawPhase1Sphere` logic. Radial gradient with specular highlight.
- **hexagon**: `Path` with 6 vertices at radius, filled with radial gradient.
- **star5/star6**: Alternating outer/inner vertices. Outer at radius, inner at radius*0.4.
- **pentagon**: 5 vertices path, radial gradient fill.
- **diamond**: Existing `_drawPhase3Diamond` logic (4-vertex rhombus).
- **gear**: Circle with N teeth (rectangular protrusions). Use `Path` with `addRect` for each tooth.
- **crescent**: Two overlapping circles subtracted. Use `Path.combine(PathOperation.difference, ...)`.
- **crown**: 3-pointed crown shape via path. Base arc + 3 triangular points.
- **cross**: Axis-aligned plus shape via `Path` with 12 vertices.
- **cube**: Isometric cube - 3 visible faces drawn as parallelograms with slightly different shades.
- **spiral**: Fibonacci spiral path drawn as thick stroke, not fill.
- **eye**: Vesica piscis (two arc intersections). Upper and lower arcs with a circle pupil.
- **snowflake**: 6 arms drawn as lines with branches at 60-degree angles.
- **bolt**: Zigzag lightning bolt path (5-7 vertices).
- **prism**: Equilateral triangle with refraction gradient.
- **leaf**: Symmetric bezier curves forming leaf shape with a center vein line.
- **ring**: Existing `_drawPhase2Ring` logic (circle with stroke style).
- **slit**: Existing `_drawPhase4Slit` logic (narrow horizontal rect).
- **singularity**: Existing `_drawPhase5Singularity` logic (black circle with aura).
- **doubleRing**: Two concentric ring strokes at different rotation speeds.
- **helix**: Two intertwined sine curves drawn in perspective.
- **tesseract**: 8 vertices of a hypercube projected to 2D, with edges connecting them.

Each method is 30-50 lines. Total approximately 700-900 lines.

**Ball effects** are applied as a wrapper around the base shape draw. In `theme_renderer.dart`:

```dart
static void drawThemedBall(Canvas canvas, VisualTheme theme, double pr,
    double glowPhase, double intensity, double stretch) {
  canvas.save();
  canvas.scale(stretch, 1.0 / stretch);

  // Apply effect transforms before drawing shape
  switch (theme.ballEffect) {
    case BallEffect.pulse:
      final scale = 1.0 + sin(glowPhase * 3 * theme.ballEffectSpeed) * 0.12;
      canvas.scale(scale, scale);
      break;
    case BallEffect.spin:
      canvas.rotate(glowPhase * 2 * theme.ballEffectSpeed);
      break;
    case BallEffect.prismatic:
      // Hue-shift the colors based on glowPhase
      // (colors are modified before passing to shape painter)
      break;
    // ... etc
  }

  // Draw glow aura
  final glowR = pr + theme.ballGlowRadius + sin(glowPhase) * 3 * intensity;
  canvas.drawCircle(Offset.zero, glowR, Paint()
    ..color = effectiveColors[0].withValues(alpha: theme.ballGlowIntensity * intensity)
    ..maskFilter = MaskFilter.blur(BlurStyle.normal, theme.ballGlowRadius));

  // Draw base shape
  BallPainters.draw(theme.ballShape, canvas, pr, effectiveColors, glowPhase, intensity);

  canvas.restore();
}
```

void _drawThemeTransition(Canvas canvas, Size size) {
  if (engine.themeTransitionTimer <= 0) return;
  final progress = 1.0 - engine.themeTransitionTimer / 1.0;

  // Expanding color ring from player
  final theme = engine.activeTheme;
  if (theme == null) return;

  final alpha = (1.0 - progress) * 0.6;
  final ringColor = theme.ballColors.first;

    Offset(engine.player.x, engine.player.y),

      ..strokeWidth = 6.0 - progress * 5.0

  );

  final washAlpha = (1.0 - progress) * 0.1;
  canvas.drawRect(
    Rect.fromLTWH(0, 0, size.width, size.height),
    Paint()..color = ringColor.withValues(alpha: washAlpha),
  );
}

## Part 6: Phase 5 Inversion Compatibility

Phase 5 inverts the background to white. All 50 themes must remain visible.

**Strategy**:
1. Each `VisualTheme` can optionally provide `invertedBallColors`, `invertedTrailColors`, `invertedObstacleColors`. If provided, these override the normal colors during phase 5.
2. If not provided, the `withInversion()` method auto-generates inverted colors by checking luminance. Any color with luminance > 0.7 (too close to white) gets its lightness reduced to 0.3.
3. Themes that are already dark (like Void Walker, Shadow Blade, Abyssal Lord) need no adjustment.
4. Themes that are light (like Arctic Frost, Phantom Flame, The Architect) need explicit inverted color overrides to maintain their identity.

**Testing approach**: Each theme must be verified at all 5 phases. The most critical check is phase 5 where the background is white.

## Part 8: DFA Logic for Post-500

After score 500, the player has seen all 5 tier themes. The DFA generates new combined themes procedurally.

### DFA Design

The DFA has states that transition based on score. Each state maps to a composite theme built by mixing components from existing themes.

```dart
class DfaThemeGenerator {
  static VisualTheme generate(int score, int dfaState, Map<int, int> rotations) {
    // Deterministic selection based on score
    final seed = score ~/ 50; // new theme every 50 points after 500

    // Select components from different tiers
    final ballTier = ((seed * 7 + 3) % 5) + 1;
    final ballVar = ((seed * 13 + 5) % 10);
    final trailTier = ((seed * 11 + 7) % 5) + 1;
    final trailVar = ((seed * 17 + 11) % 10);
    final explosionTier = ((seed * 19 + 13) % 5) + 1;
    final explosionVar = ((seed * 23 + 17) % 10);

    final ballSource = ThemeRegistry.themes[ballTier][ballVar];
    final trailSource = ThemeRegistry.themes[trailTier][trailVar];
    final explosionSource = ThemeRegistry.themes[explosionTier][explosionVar];

    // Compose a new theme from these sources
    return VisualTheme(
      id: 'dfa_$seed',
      displayName: 'Beyond ${500 + seed * 50}',
      tier: 6, // DFA tier
      variation: seed,
      ballShape: ballSource.ballShape,
      ballColors: ballSource.ballColors,
      ballGlowRadius: ballSource.ballGlowRadius,
      ballGlowIntensity: ballSource.ballGlowIntensity,
      ballEffect: ballSource.ballEffect,
      ballEffectSpeed: ballSource.ballEffectSpeed,
      trailStyle: trailSource.trailStyle,
      trailColors: trailSource.trailColors,
      trailWidth: trailSource.trailWidth,
      trailLife: trailSource.trailLife,
      trailDensity: trailSource.trailDensity,
      obstacleStyle: explosionSource.obstacleStyle,
      obstacleColors: _blendColors(ballSource.obstacleColors, trailSource.obstacleColors),
      obstacleGlowIntensity: (ballSource.obstacleGlowIntensity + trailSource.obstacleGlowIntensity) / 2,
      explosionPattern: explosionSource.explosionPattern,
      explosionColors: explosionSource.explosionColors,
      explosionParticleCount: explosionSource.explosionParticleCount,
      explosionLife: explosionSource.explosionLife,
      explosionGravity: explosionSource.explosionGravity,
      magnetStyle: trailSource.magnetStyle,
      magnetColors: trailSource.magnetColors,
      ambientStyle: ballSource.ambientStyle,
      ambientColors: ballSource.ambientColors,
      ambientCount: ballSource.ambientCount,
      wallStyle: explosionSource.wallStyle,
      wallAccentColor: explosionSource.wallAccentColor,
      wallGlowIntensity: explosionSource.wallGlowIntensity,
    );
  }

  static List<Color> _blendColors(List<Color> a, List<Color> b) {
    return [
      Color.lerp(a.first, b.first, 0.5)!,
      Color.lerp(a.last, b.last, 0.5)!,
    ];
  }
}
```

The DFA transitions every 50 points after 500 (at 550, 600, 650, etc.). Each transition triggers the same theme transition VFX (expanding ring, screen shake).

The deterministic seed-based selection ensures that the same score always produces the same composite theme, so the experience is reproducible (important for sharing/screenshots).

## Part 13: Performance Considerations

1. **Particle caps**: Each theme's `trailDensity` and `ambientCount` must be bounded. Maximum trail particles = 120 (up from 100 for themes). Maximum ambient = 40.

2. **Paint object reuse**: Create `Paint` objects once and reuse them within draw methods. Avoid allocating `Paint()` inside loops.

3. **Path caching**: For complex shapes (gear, snowflake, tesseract), compute the `Path` once when the theme activates and cache it. Scale/rotate the cached path instead of rebuilding.

4. **MaskFilter caching**: `MaskFilter.blur()` creates a new object each call. Cache frequently-used blur values.

5. **Color computation**: `withValues(alpha:)` creates a new Color object. For hot paths (trail/ambient particle loops), pre-compute the alpha-modified color once outside the loop.

6. **Conditional drawing**: If a wall style is `clean`, skip wall drawing entirely. If ambient count is 0, skip ambient drawing. Avoid unnecessary work.
